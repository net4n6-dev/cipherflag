## 1. Summary

Port EE's `v1.27.0` Layer 4 catalog hardening into CE. EE converted the three Layer 4 reference catalogs — Library EOL, Library FIPS, PQC algorithm — from hand-maintained starter sets to reproducibly-generated snapshots driven by checked-in generators at `scripts/catalogs/`, added a per-entry `Source` URL to every catalog entry, and surfaced that URL to operators via a new `model.HealthFinding.Evidence` field.

This port carries the entire feature into CE. None of it is EE-moat: the catalogs are public data and the generators are runtime-agnostic. After `v2.1.0`:

- Operators investigating an EOL (LIB-003) or FIPS (LIB-005) finding can click through to the upstream record (endoflife.date / NIST CMVP) to verify the deduction. (The PQC catalog also gains per-entry `Source` URLs for parity and invariant coverage, but no health finding consumes them in this port — matching EE's shipped scope.)
- The three catalogs become version-controlled snapshots regenerated by `make refresh-catalogs`.
- CE's existing cron `Sweeper` re-classifies stored assets on first deploy (via a `CurrentRuleEngineVersion` bump), backfilling `Evidence` on already-stored findings.

This is a **port**, not a fresh design — the architecture was decided in the EE spec (`/Users/Erik/projects/cipherflag-EE/docs/superpowers/specs/2026-05-28-layer-4-catalog-hardening-design.md`). This document specifies the CE-specific reconciliation, the divergence-driven adaptations, and the port mechanics.

### 2.2 Catalog data structures diverge in literal form

- **EOL** — CE `eolStarterMap` uses **positional 3-tuples** `{"openssl", "0.9.", "OpenSSL 0.9.x EOL 2010"}`. EE regenerated to **keyed structs** `{LibraryName:…, VersionPrefix:…, Reason:…, Source:…}`.
- **FIPS** — CE `fipsStarterMap` uses **positional 4-tuples** `{"openssl", "3.0.0", "…", "fips140-3-l1"}` (4th field = FIPS level). EE regenerated to keyed structs `{LibraryName, VersionPrefix, Note, FIPSLevel, Source}`. The `FIPSLevel` data **stays in the generated struct**; only the `LibraryFIPSLevel` *function* moved to `library_fips_lookup.go`.
- **PQC** — both use `map[string]Classification`; EE entries gain `Source`.

### 2.3 Frontend diverges structurally

CE has **no `FindingsPanel.svelte`** (the EE component the source-link rendered in). CE renders `HealthFinding` in two places:
- `frontend/src/lib/components/graph/GraphDetailPanel.svelte` (findings tab; finding card with `finding-sev`/`finding-title`/`finding-detail`/`finding-rem`)
- `frontend/src/routes/certificates/[fingerprint]/+page.svelte` (Health Findings section)

The TS type is `HealthFinding` in `frontend/src/lib/api.ts:82-91` (no `evidence` field). **CE has zero frontend test infrastructure** — no Vitest config, no test script, no existing `*.test.ts`.

## 5. Catalog reconciliation (load-bearing)

The regenerated data files must not silently drop any entry CE currently flags. Verified diffs (CE vs EE `v1.27.0`):

- **PQC** — CE 207 canonical keys ⊆ EE 208 (EE adds `secp256r1-ml-kem-768`). **No merge needed**; vendor `catalog.go` + `synonyms.go` wholesale.
- **FIPS** — CE 60 pairs vs EE 62. The 10 CE entries with empty version prefix (`aws-cloudhsm`, `azure-keyvault`, `boringcrypto`, `cryptocomply`, `dell-bsafe`, `entrust-nshield`, `microsoft-rsaenh`, `rsa-bsafe`, `thales-luna`, `yubico-yubihsm`) are all present in EE under the explicit `"*"` sentinel (confirmed at `library_fips_data.go:59,77,78,86` and others). **No real loss** — vendor wholesale, *contingent on* the `fipsVersionMatch`/`"*"` edit in §4.2 so the matcher honors `"*"`.
- **EOL** — CE 122 pairs vs EE 148. **Two genuine CE-unique entries**: `openssl 0.9.` and `openssl 1.0.`. EE's endoflife.date regen narrowed openssl to cycle-specific prefixes and dropped these still-EOL lines. **Required:** add both to `watchlists/eol.yaml`'s `manual:` overlay (`Source: "manual"`) before regenerating, so the output covers EE's 148 + these 2.

**Implementation gate:** after regenerating, re-run the `(library_name|version_prefix)` subset diff (and the PQC canonical-key diff) CE-vs-pre-port to prove zero CE entries were dropped. This gate is part of the plan's verification phase.

## 6. Testing

Ports EE's full test shape; no test hits the network (`make refresh-catalogs` is the only live-network path).

- **Catalog-invariant tests** (`internal/analysis/...`): no duplicate keys/pairs; every `Source` is `https://…` or literal `"manual"`; source URLs from allowed domains (endoflife.date / csrc.nist.gov / github.com / IETF); every PQC `Status`/`Category` is an enumerated constant; every synonym maps to a real canonical key.
- **Generator unit tests** (`scripts/catalogs/`, separate module): each `refresh-*` transform verified against captured `testdata/` fixtures. CI must invoke both `go test ./...` and `cd scripts/catalogs && go test ./...`.
- **Classifier/scorer fixture updates**: existing `library`/`classify` tests gain `Evidence` on expected findings; new tests lock the operator surface — `Check…EOL`/`Check…FIPS` findings include `source_url`; `Classify` returns `Source`.
- **Frontend (new):** Vitest component test for the source-link rendering (three cases above).

## 7. Behavioral migration / backward compat

- **API:** `Evidence` is `omitempty` — pre-`v2.1.0` clients see identical JSON.
- **DB:** **no schema migration.** `HealthFinding` serializes into the existing JSONB `findings` column; the new key rides through. (CE uses a consolidated `v2.0_baseline.sql`, not numbered migrations — irrelevant here since there is no migration.)
- **Stored findings:** existing rows have no `evidence` key → reads as `nil` → UI hides the link. Backward compatible.
- **Re-classification:** `CurrentRuleEngineVersion 4→5` marks every stored `asset_health` row stale; CE's `Sweeper` (`sweep.go:68`) rescores them in batches on first deploy, backfilling `Evidence` and flagging libraries that previously fell through to "unknown" (expanded catalog coverage). CHANGELOG documents the expected wave of new findings.
